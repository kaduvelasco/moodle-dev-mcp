# Why use moodle-dev-mcp?

AI assistants are powerful — but without context, they work in the dark. This document explains the problem that `moodle-dev-mcp` solves and when it makes sense to use it.

---

## The problem: AI without Moodle context

When you ask an AI assistant to help with Moodle development without any integration, it relies exclusively on its training knowledge — which is generic, may be outdated, and knows nothing about **your installation**.

In practice, this means:

**❌ Manual and repetitive context**
In every session, you need to copy and paste PHP files into the chat, explain the database structure, describe which plugins are installed, and specify which Moodle version is in use. Any change in the installation invalidates the context you built.

**❌ Generic and non-standard code**
Without knowing the exact Moodle version, the AI may suggest deprecated APIs, use functions that do not exist in your version, or generate code that ignores conventions specific to your installation — such as table prefixes, namespaces of existing plugins, or already defined capabilities.

**❌ Risk of regressions**
The AI does not know what already exists. It may suggest creating a table that has already been created, an event that already has a registered observer, or a capability with a conflicting name.

---

## The solution: real, automatic, and persistent context

`moodle-dev-mcp` solves these problems by exposing the real structure of your Moodle installation directly to the AI assistant via the MCP protocol.

**✅ The AI knows your installation, not a generic Moodle**
The server automatically detects the Moodle version, maps all installed plugins, reads the real database schema, and indexes the core APIs. The assistant works with real data, not assumptions.

**✅ Direct access to the database structure**
Before suggesting any query or schema change, the AI checks the real `install.xml` of your plugins. It knows which tables exist, which fields they contain, and which indexes are defined.

**✅ Scaffolding aligned with Moodle standards**
When creating new plugins, the server automatically injects the conventions of your Moodle version — correct namespaces, expected directory structure, `version.php` format, and naming patterns for capabilities and events.

**✅ Contextual code review**
When reviewing an existing plugin, the AI reads the full context generated by the server — defined functions, registered callbacks, declared dependencies — and identifies real issues instead of hypothetical ones.

**✅ Zero configuration in Moodle**
Unlike Web Services, there is no need to create API tokens, configure service users, or expose endpoints. The server reads the files directly from the local filesystem.

## Direct comparison

| Situation          | Without moodle-dev-mcp                       | With moodle-dev-mcp                                           |
| ------------------ | -------------------------------------------- | ------------------------------------------------------------- |
| Moodle version     | AI does not know — you must inform it        | Automatically detected from `version.php`                     |
| Database structure | You paste `install.xml` into the chat        | Automatically indexed in `PLUGIN_DB_TABLES.md`                |
| Core APIs          | AI uses training knowledge (may be outdated) | Indexed in `MOODLE_API_INDEX.md` with visibility and `@since` |
| Existing plugins   | You describe them manually                   | Mapped in `MOODLE_PLUGIN_INDEX.md`                            |
| Code standards     | AI uses generic PHP patterns                 | Context with Moodle and installation-specific conventions     |

---

## When to use

`moodle-dev-mcp` is ideal for:

| Scenario                            | Why the server helps                                                      |
| ----------------------------------- | ------------------------------------------------------------------------- |
| **Development of new plugins**      | Scaffold generated with real patterns from your Moodle version            |
| **Maintenance of existing plugins** | AI reads the full context before suggesting changes                       |
| **Debugging integrations**          | AI knows registered events, callbacks, and dependencies                   |
| **Code review before commits**      | Contextual analysis based on the real plugin structure                    |
| **Onboarding in legacy projects**   | AI generates a full summary of architecture, database, and execution flow |
| **Core API search**                 | Indexed search with visibility, signature, and source file                |

---

## When not to use

`moodle-dev-mcp` **is not the right tool** for:

- **Administration of Moodle instances** — to manage courses, users, and enrollments via AI, there are MCP servers based on the Moodle REST API that are complementary to this one.
- **Production environments** — the server writes context `.md` files inside plugin directories. Use it exclusively in development environments.
- **Replacing automated tests** — the server provides context to the AI but does not replace PHPUnit or Behat to ensure code correctness.
