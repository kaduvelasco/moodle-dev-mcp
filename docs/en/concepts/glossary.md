# Glossary

Technical terms used in the `moodle-dev-mcp` documentation, organized alphabetically.

---

## mtime Cache

Cache mechanism used by the Generators to avoid unnecessary regeneration of files. Before rewriting a context `.md` file, the server compares the modification date (`mtime`) of the PHP source file with the corresponding `.md` file. If the PHP file has not changed since the last generation, the file is skipped.

To force a full regeneration, ask the assistant: _"Regenerate all indexes ignoring the cache"_. The AI will call `update_indexes` with `force=true`.

→ See: [Cache System](../architecture/cache-system.md)

---

## Component

Unique identifier of a Moodle plugin in **frankenstyle** format — `type_name`. It is the value that appears in each plugin’s `version.php` and is used as a parameter in the server tools.

Examples: `local_myplugin`, `mod_quiz`, `block_html`, `auth_ldap`.

→ See: [Frankenstyle](#frankenstyle)

---

## Extractor

Internal module of `moodle-dev-mcp` responsible for reading and parsing a specific type of Moodle PHP file. Each extractor produces structured data that is consumed by the Generators.

Examples: `schema.ts` reads `db/install.xml`; `events.ts` reads `db/events.php`; `api.ts` reads `lib/*.php`.

→ See: [Extractors](../architecture/extractors.md) · [How the server works](./how-moodle-dev-mcp-works.md)

---

## Frankenstyle

Moodle naming convention used to uniquely identify plugins. The format is `type_name`, where `type` is the plugin type (`local`, `mod`, `block`, `auth`, etc.) and `name` is the plugin identifier in lowercase without hyphens.

Examples: `local_mytools`, `mod_checklist`, `block_coursestats`.

Frankenstyle is used as a table prefix in the database, as a PHP namespace, and as a parameter in the server tools (for example, the `plugin` parameter in `get_plugin_info`).

---

## Generator

Internal module of `moodle-dev-mcp` responsible for transforming the data produced by the Extractors into context `.md` files written directly into the Moodle installation. There are two types: global generators (write to the Moodle root) and plugin generators (write inside each plugin directory).

→ See: [Generators](../architecture/generators.md) · [Generated Files](../reference/generated-files.md)

---

## Hook API

Moodle extension system introduced in version 4.3 that progressively replaces the legacy callbacks from `lib.php`. It allows plugins to register to receive notifications of core events in a more structured way and with better PHP typing support.

`moodle-dev-mcp` detects and indexes both legacy callbacks and Hook API definitions in `db/hooks.php` and `classes/hook/`.

---

## MCP (Model Context Protocol)

Open standard created by Anthropic that defines how AI assistants communicate with external tools and data sources. It allows a single server to be used by any compatible client — Claude Code, Gemini Code Assist, Cursor, and others.

`moodle-dev-mcp` is an MCP server specialized in Moodle plugin development.

→ See: [What is MCP?](./what-is-mcp.md)

---

## `.moodle-mcp`

Configuration file automatically generated by the server when `init_moodle_context` is executed for the first time. It stores the Moodle installation path and the detected version, eliminating the need to pass this data in every session.

It is read by `config.ts` during server initialization. Environment variables (`MOODLE_PATH`) take priority over this file.

---

## `PLUGIN_AI_CONTEXT.md`

Main context file generated by the server inside each plugin directory. It consolidates the most relevant plugin information — architecture, database, functions, events, callbacks, and execution flow — into a single file optimized for consumption by the AI assistant.

It is the recommended entry point to start any development session on an existing plugin.

→ See: [Generated Files](../reference/generated-files.md)

---

## Prompt (MCP)

Prebuilt prompt template exposed by the MCP server that automatically injects the complete Moodle and installation context before executing a task. Unlike Tools — which execute actions — Prompts guide the AI on **how** to perform a complex task, with examples and Moodle-specific patterns.

`moodle-dev-mcp` exposes three prompts:

| Prompt            | Purpose                                        |
| ----------------- | ---------------------------------------------- |
| `scaffold_plugin` | Create the full structure of a new plugin      |
| `review_plugin`   | Review a plugin’s code with configurable focus |
| `debug_plugin`    | Debug an error with full plugin context        |

In Gemini Code Assist (Agent mode), prompts are available as slash commands: `/scaffold_plugin`, `/review_plugin`, `/debug_plugin`.

→ See: [Prompt Reference](../reference/prompts.md)

---

## Resource (MCP)

Structured data exposed by the MCP server via URI that the AI client reads passively as context — without requiring an explicit call from the user. Resources are updated whenever the context `.md` files are regenerated.

Examples of URIs: `moodle://api-index`, `moodle://plugin/local_myplugin`, `moodle://db-tables`.

→ See: [Resource Reference](../reference/resources.md)

---

## stdio

Default MCP transport mode where the server runs as a subprocess of the AI client, communicating via standard input and output (stdin/stdout). It does not open network ports. Recommended for local use when Moodle is on the same machine as the AI client.

→ See: [Architecture — Transport modes](./architecture.md#transport-modes)

---

## Streamable HTTP

Alternative MCP transport mode where the server runs as an independent HTTP service. Used when Moodle is on a different machine than the one running the AI client (e.g., remote server, isolated Docker environment). Requires authentication configuration via Bearer token.

→ See: [Architecture — Transport modes](./architecture.md#transport-modes) · [Docker & Environments](../guides/environments/docker.md)

---

## Tool (MCP)

Executable function exposed by the MCP server that the AI assistant can explicitly call to perform an action. Unlike Resources — which are read passively — Tools are invoked on demand, usually in response to a user request in natural language.

`moodle-dev-mcp` exposes 11 tools:

| Tool                      | What it does                                        |
| ------------------------- | --------------------------------------------------- |
| `init_moodle_context`     | Initializes the full installation context           |
| `generate_plugin_context` | Generates AI context for a specific plugin          |
| `plugin_batch`            | Generates context for multiple plugins at once      |
| `update_indexes`          | Regenerates the global indexes                      |
| `watch_plugins`           | Monitors plugins and updates context on save        |
| `search_plugins`          | Searches installed plugins                          |
| `search_api`              | Searches Moodle core API functions                  |
| `get_plugin_info`         | Loads the full context of a plugin into the session |
| `list_dev_plugins`        | Lists plugins under development                     |
| `doctor`                  | Diagnoses the environment and reports server health |
| `explain_plugin`          | Explains a plugin’s architecture by section         |

→ See: [Tools Reference](../reference/tools.md)
